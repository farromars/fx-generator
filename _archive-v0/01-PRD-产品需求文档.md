# 01 - 产品需求文档（PRD）

> Project DOUYIN-FX（暂定名：模型驱动短视频特效搭建平台）｜ v0.1
> 文档状态：草案 / 待评审

## 1. 背景与动机

- 短视频平台（抖音 / 剪映 / TikTok / 快手 / B 站 / 视频号 / 小红书 / Snapchat / Instagram）面部与场景特效仍以官方 SDK + 美术手工制作为主：周期长、成本高、迭代慢。
- 生成式 AI（Diffusion / GAN / Style Transfer / Face Reenactment / Lip-sync / 神经渲染）已能在端侧或离线产出风格化贴图、纹理、动画序列，但与各平台特效工程文件（DYFX / Effect House / Lens Studio / Spark AR 历史规范 / 剪映滤镜包 等）之间**没有统一的工具流**。
- 当前痛点：
  - 模型产物（贴图序列 / 3D 资产 / mask / depth / shader 参数）需要人工搬运到各平台 IDE，**重复劳动**。
  - 每个平台的工程结构、坐标系、面部 mesh 拓扑、shader 语言、性能预算各不相同，**一个特效要做 N 份**。
  - 创作者难以快速验证"模型生成的素材在目标平台真机上的效果"。
- 本项目目标是构建一个**本地工具流（Desktop Tool）**：以"模型生成"为核心生产力，以"多平台投放"为出口，让一个特效从需求 → 模型生成 → 工程化 → 多平台导出 → 真机预览 全链路在本地闭环。

## 2. 目标（Goals）

- G1：**本地优先**。所有特效工程文件、模型权重、生成素材均在本地落盘，不强依赖云端；可选云端推理用于重型模型加速。
- G2：**模型驱动生成**。集成 Diffusion / 人脸重演 / 风格化 / 抠图 / 深度估计 / 神经渲染等模型，作为特效素材的生产力源头，而非纯手绘。
- G3：**一次创作，多端投放**。同一份"特效中间表示（FX-IR）"可导出到 ≥ 2 个目标平台（v0.1 优先抖音 Effect Creator + 剪映滤镜包 / 模板）。
- G4：**节点化编辑**。提供节点图（Node Graph）作为特效搭建主界面，节点类型覆盖：模型推理节点 / 人脸跟踪节点 / 贴图节点 / shader 节点 / 时间轴节点 / 触发器节点。
- G5：**真机/虚拟摄像头预览**。本地直接看到特效跑在摄像头视频流上的效果，可录制对比；不必每次都打包上传到平台预览。
- G6：**资产可追溯**。每一个素材保留 prompt / seed / 模型版本 / 后处理参数，可一键复刻；导出包内嵌生成元数据（导出到平台时按需脱敏）。
- G7：**v0.1 闭环 demo**：用户输入一段提示词 → 本地模型生成一组面部贴图序列 → 节点图绑定到面部 landmark → 一键导出抖音 Effect Creator 工程 → 在抖音官方编辑器中可正常打开并真机生效。

## 3. 非目标（Non-Goals，明确不做）

- ✗ **不做**云端 SaaS 多人协作版本（v0.1 仅单机本地工具，多人协作进 v2 备选）。
- ✗ **不做**短视频内容创作 / 剪辑 / 发布功能（不与剪映 / CapCut 正面竞争，定位是"上游特效工厂"）。
- ✗ **不做**自研基础模型训练（v0.1 只调用开源 / 已授权的预训练模型；微调能力进 v2 备选）。
- ✗ **不做**绕过任何平台官方审核的"灰产特效"（仿冒明星脸 / 换脸 / 一键脱衣 / 政治人物等场景**直接拒绝**，详见 §11 安全合规）。
- ✗ **不做**移动端 App / Web SaaS 版本（v0.1 仅桌面端，macOS / Windows）。
- ✗ **不替代**各平台官方 IDE 的最终调试工作（导出工程后，平台审核 / 上架仍走官方 IDE，本工具只负责"生成 + 装配 + 导出"）。

## 4. 用户与场景

### 4.1 目标用户画像

| 用户类型 | 核心诉求 | v0.1 是否覆盖 |
|---|---|---|
| 独立特效创作者 | 快速产出风格化特效投稿挣分成 | ✅ 主要用户 |
| MCN / 工作室特效师 | 批量量产同主题不同变体的特效 | ✅ 次要用户 |
| 品牌方营销团队 | 节日 / 联名特效一次开发多平台投放 | ✅ 次要用户 |
| 普通用户（C 端） | 一键套模板 | ⏸ v2，v0.1 不做 |

### 4.2 核心场景

- **场景 A：风格化人脸特效**。用户输入"赛博朋克金属面具"，本地 Diffusion 模型在面部 UV 空间生成贴图，节点图绑定到 ARKit/MediaPipe 面部 mesh，导出抖音 Effect Creator 工程。
- **场景 B：动态贴纸 / 头饰**。模型生成带 alpha 通道的序列帧（如"火焰皇冠"），节点图挂到头部 anchor，时间轴循环播放，可触发"张嘴 → 火焰变大"。
- **场景 C：背景替换 + 风格迁移**。模型抠图 → 背景换成生成图 → 整体过一道风格化 LUT，导出剪映滤镜包。
- **场景 D：多平台同步导出**。同一份 FX-IR 一键产出抖音 + 剪映两份工程，用户分别上传到对应平台。

## 5. 核心概念与机制

### 5.1 FX-IR（特效中间表示）

> 整个工具的"事实源"。所有平台导出器都是 FX-IR → 平台工程的翻译器。

- 一份 FX-IR 工程 = 一个 JSON 主文件 + 一个资产目录（贴图 / 模型 / 序列帧 / shader 源码）。
- 关键字段（详见 `08-接口契约.md`，本节为概览）：
  - `version`：FX-IR schema 版本号。
  - `meta`：作者 / 创建时间 / 描述 / 目标平台清单。
  - `tracks`：跟踪锚点（face / head / hand / body / scene / segmentation_mask）。
  - `nodes`：节点图（模型推理 / 贴图 / shader / 时间轴 / 触发器）。
  - `assets`：资产清单 + 每个资产的生成元数据（prompt / seed / model_id / 后处理）。
  - `export_targets`：每个目标平台的覆写参数（分辨率上限 / 性能预算 / 不支持节点的降级策略）。

### 5.2 节点图（Node Graph）

| 节点大类 | 示例节点 | 说明 |
|---|---|---|
| **输入** | 摄像头 / 测试视频 / 静态图 | 提供视频流 / 单帧 |
| **跟踪** | 人脸 landmark / 头部姿态 / 手势 / 人像分割 | 输出锚点与 mask |
| **模型推理** | Diffusion 文生图 / Diffusion 图生图 / 风格迁移 / 人脸重演 / 抠图 / 深度估计 | 离线生成或实时推理 |
| **2D 贴图** | UV 展开 / alpha 合成 / 序列帧播放 | 把模型产物贴到锚点 |
| **3D / 渲染** | 3D 模型加载 / PBR shader / 神经渲染 | 进阶特效（v0.1 有限支持） |
| **逻辑 / 时间轴** | 触发器（张嘴 / 眨眼 / 点头）/ 定时器 / 状态机 | 交互逻辑 |
| **输出** | 预览窗口 / 录制 / 导出工程 | 终点 |

### 5.3 模型集成层（Model Hub）

- 内置常用开源模型一键下载（用户授权后从官方源拉取，不二次分发）：
  - 文生图 / 图生图：Stable Diffusion 系列（含 LoRA 加载）。
  - 抠图 / 分割：MediaPipe Selfie Segmentation、SAM。
  - 人脸：MediaPipe FaceMesh、InsightFace（仅 landmark / 不做识别 ID）。
  - 风格迁移：开源 Style Transfer 模型。
  - 深度：MiDaS。
- 推理后端可切：CPU（onnxruntime）/ Apple Silicon（Core ML / MPS）/ NVIDIA（CUDA）/ 远程 HTTP（用户自填 endpoint，例如自己的 vLLM / Replicate / 自建 ComfyUI）。
- 模型层与节点层解耦：节点只声明"我需要一个 text2img 能力"，由 Model Hub 路由到具体后端。

### 5.4 多平台导出器（Exporter）

- v0.1 优先支持：
  - **抖音 Effect Creator**（DYFX 工程）。
  - **剪映 / CapCut 滤镜包**（图像滤镜 + LUT + 简单贴纸，不含交互逻辑）。
- v1 / v2 备选：TikTok Effect House、Snapchat Lens Studio、视频号特效（若有公开 SDK）、B 站 / 快手（若有公开 SDK）。
- 每个导出器实现固定接口：`fxir_to_{platform}(fx_ir, target_options) -> 工程目录`。
- 导出时自动做能力降级：
  - 节点平台不支持 → 走预渲染（把该节点离线烘焙成贴图序列）兜底。
  - 资产超出平台尺寸 / 文件大小限制 → 自动压缩 + 警告。
  - 性能预算超标 → 在导出报告中给出优化建议（不静默修改）。

### 5.5 预览与真机回环

- 内置摄像头预览窗：直接看到节点图结果跑在本地摄像头上的效果，FPS / 推理耗时实时显示。
- 录制对比：可录制 30s 短片对比"原始视频 / 工具内预览 / 平台真机"三路差异。
- 导出后是否能在平台真机生效，仍依赖各平台官方编辑器走完打包审核流程；本工具不接管这一步，只把工程做到"打开即正确"。

## 6. v0.1 范围与里程碑

> 范围锁定原则：**先打通端到端，再加节点种类**。宁可只做 1 类特效但完整跑通"模型 → 导出 → 真机"，也不做 10 类特效但每类都断在导出环节。

### 6.1 v0.1 必做（MVP）

| 编号 | 能力 | 验收标准 |
|---|---|---|
| F1 | FX-IR schema v1 + JSON 序列化 / 反序列化 | schema 文件存在，单测覆盖核心字段 |
| F2 | 节点图编辑器（基础 UI） | 可拖拽 ≥ 8 类节点，可连线，可保存 / 加载工程 |
| F3 | 摄像头输入 + 人脸 FaceMesh 跟踪节点 | 30fps 稳定显示 landmark |
| F4 | Stable Diffusion 文生图节点（本地 / 远程二选一） | 给 prompt + seed 能产出 1 张贴图并落盘到资产目录 |
| F5 | UV 贴图绑定节点 | 把 F4 产物贴到 FaceMesh 上，预览窗实时显示 |
| F6 | 抖音 Effect Creator 导出器 | 导出工程在抖音官方编辑器中可打开、可预览、可打包 |
| F7 | 剪映滤镜包导出器（仅静态滤镜路径） | 导出滤镜包在剪映客户端可加载使用 |
| F8 | 资产元数据追踪 | 每个生成资产保留 prompt / seed / model_id |
| F9 | 安全合规拦截 | 命中黑名单（明星脸 / 政治人物 / 露点 / 违法元素）的 prompt 直接拒绝并写日志 |

### 6.2 v0.1 不做（明确推迟）

- 节点图的版本控制 / diff / merge（v2）。
- 多人协作 / 云端工程同步（v2）。
- 自研模型微调 UI（v2）。
- 移动端预览 App（v2）。
- TikTok Effect House 导出（v1）。
- 3D 模型 / 神经渲染节点（v1，v0.1 仅 2D + UV）。

### 6.3 里程碑

| 里程碑 | 内容 | 退出条件 |
|---|---|---|
| **M0：架构骨架** | FX-IR schema、模块边界、技术选型敲定 | `02-技术架构与选型.md` 评审通过 |
| **M1：本地预览闭环** | 摄像头 + FaceMesh + 静态贴图节点 + 预览窗 | 一段 demo 视频显示贴图稳定贴脸 |
| **M2：模型生成接入** | Stable Diffusion 文生图节点本地跑通 | 给定 prompt 能产出贴图并贴到脸上 |
| **M3：抖音导出闭环** | DYFX 导出器 + 抖音官方编辑器实测 | 导出工程在官方编辑器打开无报错、能打包 |
| **M4：双平台投放** | 剪映滤镜包导出器 | 同一 FX-IR 一键导出抖音 + 剪映两份工程 |
| **M5：v0.1 发布** | 文档 / 示例工程 / 黑名单 / 崩溃日志 | 内部 5 名创作者跑完场景 A，可独立产出特效 |

## 7. 系统边界与外部依赖

- **依赖各平台官方 SDK / IDE**：本工具产出"工程文件"，平台 IDE 负责打包审核，本工具不绕过审核。
- **依赖第三方模型权重**：尊重各模型的 license（CreativeML OpenRAIL-M / Apache 2.0 / 自定义），不在本工具内二次分发，仅给下载入口。
- **依赖本地算力**：默认要求 16GB 内存 + 集成显卡可跑轻量模型；Diffusion 类需 8GB 显存或走远程后端。
- **不依赖**任何专属云服务，整个工具断网仍可用（前提是模型已本地下载）。

## 8. 非功能性需求

| 维度 | 指标 |
|---|---|
| **启动时间** | 冷启动 ≤ 5s（不含模型加载） |
| **预览帧率** | 摄像头 + FaceMesh + 1 张静态贴图 ≥ 30fps（M1 Mac / 中端 PC） |
| **导出耗时** | 单平台导出 ≤ 30s（不含资产烘焙），烘焙类节点单独算 |
| **崩溃率** | 内测期单日崩溃 ≤ 1 次 / 用户 |
| **可移植性** | macOS（Apple Silicon + Intel）、Windows 10/11；Linux v0.1 不保证 |
| **可观测性** | 全链路结构化日志 + 性能 trace 可导出（用户主动触发） |

## 9. 数据与隐私

- 用户摄像头画面**仅本地处理**，不外传；如启用远程模型推理，仅上传"用户主动放进节点图的素材"，UI 上明示并需用户每次确认。
- 所有 prompt、生成素材、工程文件均落盘在用户工程目录，工具不收集、不上传。
- 崩溃日志默认只在本地，用户主动点"提交"才上传，且上传前可预览脱敏。
- 不内置任何用户行为埋点（v0.1 不做数据回流）。

## 10. 验收基线（Definition of Done）

- 一名新用户在 15 分钟内可用 v0.1 完成场景 A：输入 prompt → 生成贴图 → 贴脸 → 导出抖音工程。
- 同一份 FX-IR 工程一键导出抖音 + 剪映两份工程，两份均能在对应官方客户端正确打开。
- 黑名单 prompt 100% 被拦截，且日志可追溯（详见 §11）。
- 文档齐全：本 PRD、`02-技术架构与选型.md`、`03-任务分解与执行计划.md`、`08-接口契约.md`（FX-IR schema）、README。
- 内部 5 名创作者完成全流程，输出 ≥ 1 条可在抖音上架的特效。

## 11. 安全合规与红线（强约束）

> 本节为**强红线**，与平台审核、法律法规直接挂钩，违反则功能必须拦截，不接受"用户自负"绕过。

- **人脸相关红线**：
  - ✗ 不提供"指定真人换脸"能力（不接受上传他人照片做 face swap 的场景）；仅允许"风格化当前用户人脸"。
  - ✗ 不提供任何形式的"明星 / 政治人物 / 公众人物"识别、生成、换脸节点。
  - ✗ 不生成可被用于伪造身份证件 / 人脸验证攻击的素材（活体破解、3D 面具贴图等）。
- **内容红线**：
  - ✗ 拒绝生成色情、血腥、暴恐、政治敏感、未成年人不适宜内容；prompt 黑名单 + 生成结果二次过滤（NSFW classifier）。
  - ✗ 不规避各平台社区规范，导出器在元数据里如实标注"AI 生成"。
- **版权红线**：
  - ✗ 不内置受版权保护的 IP 素材（动漫角色 / 商标 / 影视剧截图）；用户自行上传时需确认版权归属。
  - 模型 license 在工具内显式展示，用户加载模型前需勾选确认。
- **数据合规**：
  - 摄像头权限按需申请，关闭权限时工具仍可用（仅生成路径不可用）。
  - 远程推理 endpoint 由用户自填，工具不内置任何默认上传地址。
- **审计**：
  - 所有命中红线的拦截事件写入本地日志，日志格式标准化，方便事后审计。
  - 工具自身不上报，但用户可主动导出日志配合平台调查。

## 12. 风险登记（Top Risks）

| ID | 风险 | 影响 | 缓解策略 |
|---|---|---|---|
| R1 | 抖音 Effect Creator 工程格式无公开规范 / 频繁变更 | 高 | 用版本探针检测平台 IDE 版本，导出器分版本维护；与平台官方走开发者计划对齐 |
| R2 | 本地 Diffusion 性能差，预览体验劝退 | 高 | 默认走"离线烘焙 + 缓存"，仅在编辑期跑一次；提供远程后端开关 |
| R3 | 各平台节点能力差异巨大，FX-IR 抽象漏 | 中 | v0.1 锁 2 个平台、锁 8 类节点，先把抽象做厚；新增平台时允许平台特化字段 |
| R4 | 模型 license 合规风险 | 中 | 不二次分发权重，license 文案显式展示并强制勾选 |
| R5 | 用户用工具生成违规内容连累平台 | 高 | §11 红线 + 黑名单 + NSFW 二次过滤，导出元数据标注 AI 生成 |
| R6 | 真机效果与本地预览不一致 | 中 | 导出报告附"已知差异清单"，并提供平台真机回环视频对照 |
| R7 | 工具被用作刷量 / 黑产批量生成特效 | 中 | 不提供 CLI 批量导出 API；导出频率风控（v1+） |

## 13. 待决事项（Open Questions）

> 这些问题决定 v0.1 范围，需要在 M0 评审前与产品 / 法务 / 平台关系对齐。

- Q1：v0.1 第二目标平台到底锁"剪映滤镜包"还是"TikTok Effect House"？（取决于团队是否有抖音内部资源）
- Q2：本地 Diffusion 默认是否打包权重？还是首次启动引导用户下载？（影响安装包体积与版权）
- Q3：节点图编辑器用什么技术栈实现？（详见 `02-技术架构与选型.md`）
- Q4：是否在 v0.1 内置"远程推理 endpoint"模板（如自建 ComfyUI / Replicate）？还是只留接口让用户自填？
- Q5：黑名单词库与 NSFW 模型选型由谁维护？是否可热更新？
- Q6：商业模式（v2 备选）：买断 / 订阅 / 平台分成？v0.1 不实现但需提前留接口位。
