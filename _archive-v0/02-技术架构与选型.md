# 02 - 技术架构与选型

> Project DOUYIN-FX ｜ v0.1
> 文档状态：草案 / 待评审

## 1. 设计原则

- **本地优先**：默认离线可用，远程推理作为可选加速。
- **分层解耦**：UI、节点引擎、模型推理、平台导出 四层独立演进，跨层只走稳定契约（详见 `08-接口契约.md`）。
- **Schema 驱动**：FX-IR 是事实源，UI 与导出器都围绕它构建；schema 变更走版本号 + 迁移器，不允许破坏性修改。
- **不绑死单一模型 / 单一平台**：模型走 `ModelProvider` 接口，平台走 `Exporter` 接口，新增成员**只加文件，不改框架**。
- **能离线烘焙就不要实时**：性能预算让位给体验，重型节点（Diffusion / 风格迁移）默认离线烘焙落盘，编辑期只取缓存。
- **拒绝隐式行为**：能力降级、资产压缩、prompt 拦截 必须在 UI 与导出报告中显式提示，不静默吞掉。

## 2. 总体架构（分层）

```
┌─────────────────────────────────────────────────────────────┐
│  UI 层（Desktop App）                                        │
│   - 节点图编辑器 / 资产面板 / 预览窗 / 导出向导               │
└──────────────┬──────────────────────────────────────────────┘
               │ 调用
               ▼
┌─────────────────────────────────────────────────────────────┐
│  Core 引擎层                                                 │
│   - FX-IR 模型 + 序列化 / 迁移                                │
│   - 节点图执行器（DAG 调度 / 缓存 / 增量计算）                 │
│   - 资产管理（落盘 / 元数据 / 去重）                          │
│   - 安全合规中间件（prompt 黑名单 / NSFW 过滤）               │
└──────┬──────────────────────────┬───────────────────────────┘
       │                          │
       ▼                          ▼
┌──────────────────┐       ┌──────────────────────────────┐
│ Model Hub        │       │ Exporter Hub                 │
│ - SD / FaceMesh  │       │ - DouyinEffectCreator        │
│ - SAM / MiDaS    │       │ - JianyingFilterPack         │
│ - 风格迁移       │       │ - TikTokEffectHouse (v1)     │
│ 后端路由：       │       │ - LensStudio (v2)            │
│   CPU / MPS /    │       └──────────────────────────────┘

```

- **依赖方向**：UI → Core → (Model Hub / Exporter Hub)，下层不反向依赖上层。
- **进程模型**：v0.1 单进程；模型推理重任务走子进程 / 线程池，避免阻塞 UI（详见 §6）。

## 3. 模块边界

| 模块 | 职责 | 不做的事 |
|---|---|---|
| `app/`（UI） | 渲染节点图、资产面板、预览、导出向导 | 不做模型推理 / 不做导出格式细节 |
| `core/fxir/` | FX-IR schema、序列化、版本迁移 | 不依赖 UI / 不依赖模型 |
| `core/graph/` | DAG 调度、节点执行、缓存、增量计算 | 不实现具体节点逻辑 |
| `core/assets/` | 资产落盘 / 去重 / 元数据 / GC | 不做生成 |
| `core/safety/` | Prompt 黑名单、NSFW 过滤、审计日志 | 不做业务决策 |
| `nodes/` | 具体节点实现（输入 / 跟踪 / 模型 / 贴图 / 输出） | 不直接读写 FX-IR |
| `models/` | ModelProvider 接口 + 各模型实现 + 后端路由 | 不感知节点 |
| `exporters/` | 各平台导出器实现 | 不感知 UI |
| `cli/`（v1+） | 离线 / 批处理入口（v0.1 不开放公开 API，仅内测调试） | — |

## 4. 技术栈选型

### 4.1 候选对比

| 方案 | 优点 | 缺点 | v0.1 选择 |
|---|---|---|---|
| **Tauri + React + TypeScript（前端）+ Rust（壳）+ Python（推理子进程）** | 安装包小（~10MB），UI 生态成熟，Rust 壳安全；Python 子进程接 PyTorch / ONNX 生态 | 需要维护 Rust + TS + Python 三栈；跨进程通信开销 | ✅ **首选** |
| Electron + Node 全栈 | 单一 JS 栈，生态最大 | 安装包动辄 100MB+，模型推理需绕 Node 调 Python 或走 ONNX.js（性能差） | ❌ |
| PyQt / PySide6 全 Python | 单一栈，直接调 PyTorch | UI 现代感弱，节点图编辑器需自实现，跨平台打包重 | ❌ |
| Flutter Desktop | UI 现代，跨平台一致 | 节点图组件生态薄，Dart 调 Python 链路长 | ❌ |
| Web 版（浏览器内 + ONNX.js / WebGPU） | 零安装 | 摄像头权限受浏览器限制；本地文件系统访问差；模型权重每次拉取 | ❌（与"本地优先"冲突） |

### 4.2 v0.1 锁定栈

| 层 | 技术 | 备注 |
|---|---|---|
| 桌面壳 | Tauri 2.x | Rust 主进程 + WebView UI |
| UI | React 18 + TypeScript + Vite | 主流前端栈 |
| 节点图组件 | React Flow（MIT） | 成熟，社区活跃 |
| 状态管理 | Zustand | 轻量，避免 Redux 模板代码 |
| 推理子进程 | Python 3.11 + onnxruntime / PyTorch | 通过 stdio JSON-RPC 与 Rust 主进程通信 |
| 模型框架 | PyTorch（开发期）+ ONNX（发布期优先） | 能转 ONNX 的模型一律转，减少 PyTorch 依赖体积 |
| 视频 / 图像 | OpenCV（Python）+ ffmpeg（Rust 调 CLI） | 摄像头采集 / 编解码 |
| 序列化 | JSON（FX-IR）+ MessagePack（节点间大对象） | 工程文件人类可读 |
| 日志 | tracing（Rust）+ structlog（Python）| 统一结构化 JSON |
| 打包 | Tauri bundler + signtool / notarytool | macOS 公证、Windows 签名 |

### 4.3 依赖红线

- 不引入 GPL / AGPL 库（避免污染商业化路径）。
- 不引入需要联网激活的闭源 SDK。
- 模型权重不打进安装包，首次启动引导下载（见 §7）。

## 5. FX-IR 数据模型（概览）

详细 schema 见 `08-接口契约.md`，本节仅给关键决策。

- **顶层结构**：`{ version, meta, tracks, nodes, edges, assets, export_targets }`
- **版本策略**：
  - schema 主版本号变更 → 走迁移器（`core/fxir/migrate/`），永不破坏旧工程。
  - 节点新增属性 → 走 `default` 字段，旧工程加载时填默认值。
- **资产寻址**：所有资产用内容哈希命名（sha256-prefix），工程 JSON 只引用哈希；节省空间 + 天然去重。
- **节点端口类型**：强类型（`Image / Mask / Mesh / Float / Vec / Trigger / Asset`），UI 层在连线时校验，引擎层在执行时再校验一次。

## 6. 节点引擎执行模型

- **DAG 调度**：节点图编译为 DAG，拓扑排序后执行；环检测在编辑期实时反馈。
- **执行模式**：
  - **预览模式**：实时跑摄像头流，每帧执行 DAG；重型节点（Diffusion）走"上次缓存"，不每帧重算。
  - **烘焙模式**：用户主动触发某节点 / 某子图烘焙；产物落盘成贴图序列或单图，下游节点引用烘焙结果。
  - **导出模式**：执行一次完整 DAG，把所有需要打包的资产固化，交给 Exporter。
- **缓存键**：`hash(节点类型 + 参数 + 上游输入哈希 + 模型版本)`；只要键不变，复用磁盘缓存。
- **增量计算**：参数变更只标脏受影响子图，不整图重算。
- **错误传播**：单节点异常不崩整图，UI 标红并展示堆栈；导出时若关键节点异常，导出器拒绝出包。

## 7. 模型集成与后端路由

### 7.1 ModelProvider 接口

```python
class ModelProvider(Protocol):
    capability: str          # "text2img" / "img2img" / "facemesh" / "matting" / ...
    backend: str             # "cpu_onnx" / "mps" / "cuda" / "remote_http"

    def warm_up(self) -> None: ...
    def infer(self, payload: dict) -> dict: ...
    def release(self) -> None: ...
```

- 节点只声明"我需要 capability=text2img"，由 Model Hub 按用户偏好 + 硬件可用性路由到具体 Provider。
- 路由优先级（默认）：`cuda > mps > cpu_onnx > remote_http`，可由用户在设置里覆写。

### 7.2 内置模型清单（v0.1）

| Capability | 模型 | License | 默认后端 |
|---|---|---|---|
| facemesh | MediaPipe FaceMesh | Apache 2.0 | cpu_onnx |
| matting | MediaPipe Selfie Segmentation | Apache 2.0 | cpu_onnx |
| matting+ | SAM (ViT-B) | Apache 2.0 | mps / cuda |
| text2img / img2img | Stable Diffusion 1.5 / SDXL Turbo | CreativeML OpenRAIL-M | mps / cuda / remote |
| style_transfer | 开源 fast-style-transfer | MIT | cpu_onnx |
| depth | MiDaS small | MIT | cpu_onnx |

- 权重首次启动引导下载，**从模型官方源**（HuggingFace / GitHub Release / 各模型官网），工具不二次分发。
- License 在工具内显式展示，用户加载前需勾选确认（写入本地 `model_consent.json`，不上传）。

### 7.3 远程推理后端

- 用户自填 endpoint（OpenAI 兼容 / ComfyUI / 自建 vLLM 等），工具不内置默认地址。
- 上传前 UI 显式提示"以下数据将上传到 {endpoint}"，需用户每次确认（可勾选"本工程内不再提示"）。
- 远程后端失败自动降级到本地后端，不静默切换。

## 8. 多平台导出器架构

### 8.1 Exporter 接口

```python
class Exporter(Protocol):
    platform: str            # "douyin_effect_creator" / "jianying_filter" / ...
    target_versions: list[str]   # 支持的平台 IDE 版本

    def capabilities(self) -> ExportCapabilities: ...
    def export(self, fx_ir: FXIR, options: ExportOptions) -> ExportResult: ...
```

- `capabilities()` 声明该平台支持的节点类型 / 资产格式 / 性能预算。
- `export()` 把 FX-IR 翻译为平台工程目录，不可识别的节点走能力降级（见 §8.3）。

### 8.2 v0.1 / v1 / v2 路线图

| 平台 | 阶段 | 备注 |
|---|---|---|
| 抖音 Effect Creator (DYFX) | v0.1 | 首要目标 |
| 剪映 / CapCut 滤镜包 | v0.1 | 仅静态滤镜 + LUT 路径 |
| TikTok Effect House | v1 | 与抖音同源但工程结构不同 |
| Snapchat Lens Studio | v2 | JS 脚本引擎，门槛高 |
| Spark AR | — | 已停止维护，不做 |
| 视频号 / 快手 / B 站 | v2 备选 | 取决于公开 SDK 进展 |

### 8.3 能力降级策略

| 情况 | 策略 |
|---|---|
| 节点类型平台不支持 | 上游烘焙为贴图序列后注入 |
| 资产分辨率超平台上限 | 自动等比缩放 + 警告 |
| 总包大小超限 | 报告中列出 Top N 大资产 + 建议 |
| 性能预算超标（每帧推理时长） | 报告中给出"建议改用烘焙节点"提示，**不静默修改**用户工程 |

## 9. 安全合规中间件

详见 `07-安全合规与红线清单.md`，本节仅给架构位置。

- **三道关卡**：
  1. **Prompt 入口**：text2img / img2img 节点提交前，prompt 过黑名单 + 关键词分类器。
  2. **生成结果**：Diffusion 出图后过 NSFW classifier；命中则丢弃 + 审计日志。
  3. **导出前**：导出器扫一遍工程内所有图像资产，再过一次 NSFW classifier。
- 命中即拒，不"用户自负"绕过；审计日志只存本地，不上报。

## 10. 性能与资源预算

| 场景 | 目标 | 兜底 |
|---|---|---|
| 摄像头 + FaceMesh + 1 张静态贴图 | ≥ 30fps | 不达标则提示降低预览分辨率 |
| Diffusion 文生图（SDXL Turbo / mps） | 单图 ≤ 5s | 走远程后端 |
| 工程加载 | ≤ 2s（不含模型） | 大资产懒加载 |
| 内存占用（空闲） | ≤ 1.5GB | — |
| 内存占用（运行 SDXL Turbo） | ≤ 8GB | — |

## 11. 可观测性

- **日志**：结构化 JSON，分级 `trace/debug/info/warn/error`，默认 info+；用户可在设置里开 trace。
- **性能 trace**：每帧 DAG 执行耗时、各节点耗时、缓存命中率，可导出 chrome trace 格式。
- **崩溃报告**：本地落盘，用户主动点"提交"才上传（v0.1 不内置上传后端，留接口）。
- **指标**：v0.1 不做埋点，仅提供本地查看。

## 13. 已知风险与对策（架构层面）

| 风险 | 对策 |
|---|---|
| Tauri 在某些 Linux 桌面环境兼容性差 | v0.1 仅承诺 macOS + Windows |
| Python 子进程启动慢 | 应用启动后预热常驻，避免每次冷启 |
| 模型权重下载因网络受限失败 | 提供镜像列表 + 手动导入入口 |
| FX-IR schema 频繁变更 | 主版本变更必带迁移器 + 单测覆盖；CI 每次合并跑历史工程回归 |
| 平台 IDE 工程格式逆向不稳 | 与平台官方开发者计划对齐；导出器分版本维护 |
